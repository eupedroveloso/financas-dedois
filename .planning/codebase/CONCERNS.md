# Codebase Concerns

**Analysis Date:** 2026-05-16

## Tech Debt

**Unedited boilerplate page:**
- Issue: `app/page.tsx` is the verbatim Create Next App starter template. It renders placeholder content (Vercel deploy links, Next.js logo, template text) and contains no project-specific logic. The entire file needs to be replaced before any real feature work begins.
- Files: `app/page.tsx`
- Impact: The app has no actual functionality. All downstream feature work depends on this file being cleaned up first.
- Fix approach: Replace the file contents entirely with the root page UI for the project.

**Boilerplate metadata in layout:**
- Issue: `app/layout.tsx` uses the default `title: "Create Next App"` and `description: "Generated by create next app"` metadata values. These will appear in browser tabs, search results, and social previews.
- Files: `app/layout.tsx` (lines 15-18)
- Impact: Poor SEO and confusing UX in production.
- Fix approach: Update the `metadata` export with the real app name and description before deployment.

**Empty `next.config.ts`:**
- Issue: `next.config.ts` contains no configuration beyond the empty object `const nextConfig: NextConfig = {}`. As the project grows, missing config (CSP headers, image remote patterns, redirects, etc.) will require retroactive changes.
- Files: `next.config.ts`
- Impact: No security headers, no image domain allow-list, no custom rewrites. These omissions may cause issues when adding features.
- Fix approach: Add `headers()` for Content-Security-Policy, configure `images.remotePatterns` when external images are needed.

**No Node.js version pin:**
- Issue: There is no `.nvmrc`, `.node-version`, or `engines` field in `package.json`. Next.js 16 requires Node.js 20.9.0 or higher. The environment runs Node.js 22.20.0, which satisfies this, but without a pin any collaborator or CI environment could use an incompatible version silently.
- Files: `package.json`
- Impact: Inconsistent environments; potential runtime failures on Node.js < 20.9.
- Fix approach: Add `"engines": { "node": ">=20.9.0" }` to `package.json` and create a `.nvmrc` set to `22`.

## Known Bugs

**Font declaration conflict in `globals.css`:**
- Symptoms: `body` in `app/globals.css` declares `font-family: Arial, Helvetica, sans-serif` (line 25), but the Tailwind theme maps `--font-sans` to the Geist font loaded in `layout.tsx`. The Tailwind utility `font-sans` will use Geist, but the bare `body` CSS rule overrides it with Arial. Any element that does not explicitly apply `font-sans` via Tailwind will render in Arial, not Geist.
- Files: `app/globals.css` (line 25), `app/layout.tsx` (lines 5-13)
- Trigger: Renders on any element not carrying the `font-sans` Tailwind class.
- Workaround: Explicitly apply `font-sans` on the `<body>` or remove the conflicting `font-family: Arial` declaration from `globals.css`.

## Security Considerations

**No Content Security Policy:**
- Risk: The app sends no CSP headers. When client-side JavaScript is added (third-party scripts, analytics, embeds), there is no mitigation against XSS injection.
- Files: `next.config.ts`
- Current mitigation: None. The app has no JavaScript-heavy pages yet, reducing immediate risk.
- Recommendations: Add a `headers()` function in `next.config.ts` that sets a strict `Content-Security-Policy` before launching.

**External links lack `rel="noopener noreferrer"`... partially:**
- Risk: `app/page.tsx` has two `<a>` tags with `target="_blank"`. Both correctly set `rel="noopener noreferrer"`. However, this is only present in the boilerplate and must be enforced as a convention as real links are added.
- Files: `app/page.tsx` (lines 40-43, 53-56)
- Current mitigation: The ESLint rule `react/jsx-no-target-blank` is included via `eslint-config-next/core-web-vitals`, which will warn when this is omitted.
- Recommendations: Confirm the ESLint rule is active by running `eslint` and keep the lint step in CI.

## Performance Bottlenecks

**Google Fonts loaded at runtime (network-dependent):**
- Problem: `layout.tsx` loads Geist Sans and Geist Mono via `next/font/google`. In development this makes a network request to Google's font CDN. In production Next.js self-hosts these fonts, which is correct — but if the initial build environment lacks internet access (air-gapped CI), the build will fail.
- Files: `app/layout.tsx` (lines 5-13)
- Cause: `next/font/google` fetches and bundles fonts at build time.
- Improvement path: Use `next/font/local` with font files committed to `public/fonts/` if build environment internet access is not guaranteed.

## Fragile Areas

**Single-file `app/` structure:**
- Files: `app/layout.tsx`, `app/page.tsx`, `app/globals.css`
- Why fragile: The entire app lives in three files with no sub-directories, no components directory, no utilities. Any new route, component, or shared logic added without establishing a directory convention first will create inconsistent structure that is expensive to refactor later.
- Safe modification: Establish `app/components/`, `app/lib/`, and `app/(routes)/` directories before adding any second page or shared component. See `node_modules/next/dist/docs/01-app/01-getting-started/02-project-structure.md` for the recommended layout.
- Test coverage: Zero — no test files exist anywhere in the project.

## Scaling Limits

**No database, auth, or API layer:**
- Current capacity: Static single-page app.
- Limit: The project has no backend infrastructure. Adding any data-driven feature requires selecting and integrating a database, an auth provider, and API routes from scratch.
- Scaling path: Add Route Handlers under `app/api/` for backend logic; add an ORM (e.g., Prisma, Drizzle) and a database provider before any data persistence work.

## Dependencies at Risk

**Next.js 16 breaking changes not yet relevant but will be when features are added:**
- Risk: Next.js 16 removed synchronous access to `cookies()`, `headers()`, `draftMode()`, and `params`/`searchParams` in pages. The single-argument form of `revalidateTag()` is deprecated. The `middleware` filename convention is deprecated (renamed to `proxy`). None of these are used yet, but any developer who follows Next.js 14/15 patterns from memory or training data will introduce code that does not compile or behaves incorrectly.
- Impact: Silent runtime failures or TypeScript errors when adding auth, caching, or routing features.
- Migration plan: Reference `node_modules/next/dist/docs/01-app/02-guides/upgrading/version-16.md` before implementing any of the above APIs. The project's `AGENTS.md` already mandates reading these docs.

**React 19.2 (canary features):**
- Risk: The App Router in Next.js 16 uses React 19.2 canary features (View Transitions, `useEffectEvent`, `Activity`). These APIs are not yet stable React releases. External libraries that peer-depend on a specific stable React version may emit warnings or fail.
- Impact: Third-party component libraries installed later may conflict with React 19.2 canary.
- Migration plan: Check peer dependency compatibility of any UI library before installing.
